1. 요약

SH와 LH 모두 최종 산출물을 공고 원문 덤프가 아니라 QA 가능한 정제 데이터로 수렴하도록 개선했다. 최종 저장 단위는 신청 가능한 단위인 Offering이며, 동호수가 공개되지 않는 모집 방식도 처리할 수 있도록 unit_no는 nullable 전제로 유지했다.

393 notice schedules

0 pending offerings after QA

2. 배경

  • 기존 SH 수집은 재수집 시 source_span에 로컬 object root 경로가 들어가 idempotency가 깨졌다.
  • 동일 공고/첨부에서 approved 15건과 pending 15건이 같은 신청 단위로 중복되어 QA 승격 시 unique 충돌이 발생했다.
  • PDF 추출은 일부 하드코딩 패턴에 치우쳐 있었고, HTML/HWP/HWPX 첨부 및 미리보기 보존 흐름이 부족했다.
  • LH는 SH 게시판과 별개로 MyHome OpenAPI 기반의 독립 수집 경로가 필요했다.
  • Serving/API 기본 노출은 raw archive가 아니라 QA-approved normalized offerings만 대상으로 해야 했다.

3. 목표 모델

원본 보존 공고/첨부를 object storage 인터페이스로 저장
Artifact 추출 PDF, XLSX, HTML, HWPX, HWP 기계 추출
정규화 신청 가능한 단위인 Offering과 schedule 생성
LLM 보정 deterministic parser 실패 구간만 repair
QA 승인 approved 데이터만 serving/API 기본 노출

source_span은 object://... 또는 myhome://... 같은 논리 원천 식별자를 사용한다. .data/... 등 로컬 경로 기반 span은 QA-approved로 승격하지 않는다.

4. 구현 내역

SH 파이프라인

  • 첨부 원본과 preview HTML을 보존하고, 추출 artifact의 source span을 object key 기반으로 안정화했다.
  • 공급 주택 팸플릿처럼 파일명만으로 누락되던 공급 정보성 PDF를 notice_pdf 계열로 분류하도록 보강했다.
  • PDF/XLSX artifact 재처리 시 같은 입력에서 중복 offering이 승인되지 않도록 QA와 upsert 기준을 정리했다.
  • SH 본문/첨부에서 신청기간을 schedule 테이블로 구조화하는 경로를 추가했다.

LH/MyHome 파이프라인

  • rsdtRcritNtcList와 ltRsdtRcritNtcList를 사용하는 독립 수집 명령을 추가했다.
  • pblancId, suplyInsttNm, suplyTyNm, fullAdres, hsmpNm, sumSuplyCo, rentGtn, mtRntchrg, beginDe, endDe를 offerings/schedules에 매핑했다.
  • 기관 필터는 응답 필드 suplyInsttNm 기준으로 클라이언트에서 적용한다.
  • 최신 공개 수집 경로는 공공데이터포털 MyHome 모집공고 API를 주 경로로, MyHome 단지정보/예비입주자 API를 보조 경로로 정리했다.

추출 및 LLM

  • HTML preview, HWPX zip/xml text, HWP 외부 도구 기반 추출 및 hwp_unsupported fallback을 추가했다.
  • LLM repair는 .env 키를 사용하고, 전체 시도 횟수는 최대 1,500회로 제한했다.
  • LLM 결과가 한 artifact에서 여러 offering을 생성할 때 #llm_item=N suffix로 source span 충돌을 방지했다.
  • llm candidates 기본 조회는 이미 approved offering이 있는 공고를 제외한다.

API/QA

  • /offerings 기본 응답은 qa_status=approved만 반환한다.
  • /schedules API를 추가하고, approved offering이 있는 공고의 일정만 노출한다.
  • QA 승격은 pending뿐 아니라 전체 record를 다시 평가해 stale approved 상태를 남기지 않도록 했다.

8. 이슈와 해결

이슈 원인 처리
SH 재수집 unique 충돌 source_span에 .data/objects와 .data/objects-check 같은 로컬 경로가 들어감 object://... 논리 span으로 교체하고 QA에서 legacy local-path span 승인 거부
pending 중복 후 QA 승격 충돌 approved partial unique index만으로 pending 중복을 막지 못함 QA 재평가와 application-unit/source identity 기준 정리
공급 정보 첨부 누락 파일명 기반 classifier가 팸플릿/공급 정보 PDF를 unsupported로 처리 공급 정보성 PDF 분류 보강
LH API 키 환경 셸에 MYHOME_API_KEY가 export되지 않으면 CLI가 즉시 실패 검증 명령은 --service-key로 실행. 문서에는 키 값 대신 보관 방식만 설명

9. 남은 권장 작업

  • 마이홈 단지정보 API를 offering enrichment 후보로 붙이는 작업은 별도 이터레이션에서 진행한다.
  • HWP 외부 추출 도구 설치 여부를 운영 환경 체크 항목에 추가한다.
  • LLM repair 결과는 artifact별 회귀 fixture를 축적해 deterministic parser 개선 재료로 사용한다.
  • SH/LH 실시간 API 수치는 변동 가능하므로 배포 전 dry-run으로 재확인한다.
