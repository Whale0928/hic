집 모아 / House Information Collector

SH/LH 공공주택 공고와 첨부파일을 개별 주택 정보로 정규화하기 위한 현재 구현 흐름과 목표 설계 가이드

작성일 2026-05-14

현재 구현 흐름

SH 주택임대 게시판 m_247, multi_itm_seq=2만 수집. 주택분양은 아직 미연결.
목록/상세 HTML 파서 공고 seq, 제목, 부서, 등록일, 조회수, 본문, 첨부 메타를 파악.
Raw Archive 중심 저장 raw_documents, attachments, attachment_extractions에 원문과 추출 행을 보존.
XLSX/PDF Extractor XLSX 전체 행 저장, 주택목록으로 보이는 행만 housing_units 후보 생성. PDF plain text 저장.
현재 설계 리스크 raw 보존은 잘 되지만, 서비스가 원하는 개별 정보 모델이 아직 얇다. 공고/결과/알림 범위도 더 명확히 분리해야 한다.

목표 설계 흐름

Board Registry SH 주택임대, SH 주택분양, 이후 LH를 모두 설정 기반으로 등록.
Notice Classifier 모집공고, 정정공고, 결과발표, 계약안내, 일반알림을 분리. 기본 서비스 대상은 모집/정정공고.
Attachment Classifier 주택목록, 공고문, 공급현황, 평면도, 신청서, 당첨자명단, 기타로 첨부 의미를 먼저 분류.
Domain Normalizer 단지, 주소, 호수, 면적, 가격, 자격, 일정, 공급유형을 명시적 테이블로 저장.
Raw Archive는 보조 계층 재처리와 감사용으로 원문은 계속 저장하되, 조회/API의 중심은 정규화 테이블로 이동.

공고만 수집한다는 기준

수집 대상

모집공고, 정정공고, 추가모집, 잔여세대 공급처럼 실제 주택 선택/신청에 필요한 게시물.

보조 저장 대상

당첨자 발표, 계약안내, 시스템 알림은 기본 서비스 데이터에서 제외하고 raw/archive 또는 이벤트로만 저장.

제외 대상

개인정보성 당첨자 명단 XLSX는 호수 데이터 후보로 만들지 않고 추출 원문만 관리.

권장 기본값

  • collection_scope = recruitment_only를 기본값으로 둔다.
  • 운영/감사용 원문 보존이 필요하면 archive_non_recruitment = true를 별도 옵션으로 둔다.
  • API와 대시보드는 모집/정정공고 기반의 offers만 기본 노출한다.

주택분양 추가 위치

주택임대

m_247, multi_itm_seq=2. 현재 구현되어 있고 XLSX 호수 추출까지 검증됨.

주택분양

m_244, multi_itm_seq=1. 다음 우선순위로 같은 Board Registry에 추가.

LH 확장

게시판/첨부 구조가 달라도 Normalizer 이후 도메인 테이블은 공유한다.

설계 포인트

  • 코드에 RentalListPath처럼 박아두지 말고 BoardConfig로 일반화한다.
  • 게시판별 수집 결과를 board_kind와 agency로 명확히 구분한다.
  • 주택분양은 가격/면적/분양일정 필드가 임대와 다를 수 있으므로 offer_type을 둔다.

다음 설계 가이드에서 결정할 질문

1. raw 저장 범위

공고 외 게시물도 raw로 보존할지, 아니면 모집공고만 수집 자체를 할지 결정해야 한다.

2. 개별 정보의 기준

호수 단위만 개별 정보인지, 단지/타입/공급세대수 단위도 개별 정보로 볼지 정해야 한다.

3. LLM 정제 지점

첨부 원문 전체를 LLM에 넣기보다, 표 후보 행과 낮은 confidence 필드만 보정하는 방식이 비용과 품질 면에서 적절하다.
